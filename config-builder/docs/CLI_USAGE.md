# Command-Line Usage

The config-builder ships with a CLI that exposes the same data the
GUI uses, so you can render profiles and validate cfgs from scripts,
Ansible playbooks, CI pipelines, or just a one-off shell command.

## Invoking the CLI

After installing via the standalone binary or `pip install`, the
command is `vanguard-config-builder`:

```bash
vanguard-config-builder COMMAND [OPTIONS]
```

If you're running from a checkout without installing:

```bash
python -m vanguard_config_builder COMMAND [OPTIONS]
```

Without a command the GUI launches — same behavior as v0.1.0's
double-click flow.

## Subcommand overview

| Command | Purpose |
|---|---|
| `generate` | Render a profile to a `server.cfg` file |
| `validate` | Check an existing `server.cfg` for problems |
| `list-profiles` | Print the bundled profiles |
| `list-cvars` | Print the known cvar database |
| `diff` | Compare two `server.cfg` files cvar-by-cvar |

`--help` works at every level:

```bash
vanguard-config-builder --help
vanguard-config-builder generate --help
```

## Exit codes

| Code | Meaning |
|---|---|
| `0` | Success |
| `1` | Usage error (argparse output, e.g. missing required flag) |
| `2` | Validation issues found — by `validate`, or by `generate --strict` |
| `3` | IO / configuration error — file not found, profile not found, unwritable path |

CI pipelines should treat `2` as "config has problems, fix it" and
`3` as "infrastructure issue, retry won't help".

## `generate` — render a profile

```text
vanguard-config-builder generate --profile NAME -o PATH
                                 [--no-header]
                                 [--strict | --no-strict]
```

Resolves PROFILE through its `extends:` chain, validates the result
against the cvar schema (unless you opt out), and writes the rendered
cfg to OUTPUT.

| Flag | Default | Description |
|---|---|---|
| `--profile NAME` | required | Profile filename stem (e.g. `cup`, `public`, `scrim`). Run `list-profiles` to see what's available. |
| `-o`, `--output PATH` | required | Where to write the cfg. The file is overwritten if it exists. |
| `--no-header` | header on | Skip the `// Generated by ...` metadata block at the top. Use for diff-friendly outputs. |
| `--strict` / `--no-strict` | strict on | Fail with exit `2` if validation finds issues. With `--no-strict` the file is written anyway and a warning goes to stderr. |

### Examples

Render the bundled Cup profile and write to `/etc/vanguard/server.cfg`:

```bash
vanguard-config-builder generate --profile cup -o /etc/vanguard/server.cfg
```

Render without the metadata header (useful for cfgs you check into git):

```bash
vanguard-config-builder generate --profile public -o public.cfg --no-header
```

Force-write even if validation complains (rare; usually the validator
is right):

```bash
vanguard-config-builder generate --profile experimental \
                                 -o /tmp/experimental.cfg --no-strict
```

Exit codes:
- `0` — file written, no validation issues (or `--no-strict` set)
- `2` — `--strict` found validation issues; nothing was written
- `3` — profile not found, unwritable output path, broken cvar database

## `validate` — check an existing cfg

```text
vanguard-config-builder validate FILE [--format text|json]
```

Parses FILE and reports cvar-schema and cross-cvar issues:

- Unknown cvars (not in the database)
- Type mismatches (`sv_fps "abc"`)
- Out-of-range integers
- Invalid enum values
- Cross-cvar warnings (e.g. `g_warmup` set with `g_doWarmup=0`)

| Flag | Default | Description |
|---|---|---|
| `--format text` | text | One issue per line, plain prose. |
| `--format json` | | Machine-readable: `{"file": ..., "valid": ..., "issues": [...], "cvar_count": N}`. |

### Examples

Quick check of an in-place config:

```bash
vanguard-config-builder validate /etc/vanguard/server.cfg
```

Output on a clean file:

```text
/etc/vanguard/server.cfg: OK (24 cvars)
```

Output on a file with issues:

```text
/etc/vanguard/server.cfg: 2 issue(s):
  - sv_fps: value 9999 out of range [20..125]
  - g_warmup is set but g_doWarmup=0 — warmup will be ignored
```

Pipe JSON output into `jq` for tooling:

```bash
vanguard-config-builder validate server.cfg --format json | jq '.issues'
```

Exit codes:
- `0` — no issues found
- `2` — at least one issue found
- `3` — file not found / unreadable

## `list-profiles`

```text
vanguard-config-builder list-profiles
```

Prints the user-visible profiles bundled with the tool: filename
stem, display label, description, inheritance chain, and resolved
cvar count. Profiles starting with `_` (abstract bases) are not
listed.

### Example

```text
$ vanguard-config-builder list-profiles
cup         Cup / Tournament
            ESL-ready competitive setup (40fps, 6v6, strict class limits)
            (chain: cup <- _base, 19 cvars)
public      Public Server
            32-slot pub setup (20fps, no friendly fire, unlimited classes)
            (chain: public <- _base, 19 cvars)
scrim       Practice / Scrim
            Fast practice loop with diagnostics enabled
            (chain: scrim <- _base, 19 cvars)
```

Exit code: always `0` unless a profile YAML is broken.

## `list-cvars`

```text
vanguard-config-builder list-cvars [--category ID] [--format text|json]
```

Prints every cvar known to the tool. Use `--category` to narrow by
category (`identity`, `network`, `match`, `hitbox`, `anticheat` —
whatever your `cvars.yaml` declares).

| Flag | Description |
|---|---|
| `--category ID` | Show only cvars in this category. |
| `--format text` (default) | Aligned columns: name, type, category, default, optional `[SECRET]` marker. |
| `--format json` | Array of `{name, type, default, category, description, archive, secret}` objects. |

### Examples

```bash
$ vanguard-config-builder list-cvars --category network
sv_fps                          int       network     default=20
sv_maxclients                   int       network     default=32
g_antilag                       bool      network     default=1
g_antiwarp                      bool      network     default=1
vanguard_netcode_profile        enum      network     default='public'
```

Generate a documentation table from JSON:

```bash
vanguard-config-builder list-cvars --format json \
  | jq -r '.[] | [.name, .type, .description] | @tsv' \
  > cvars.tsv
```

Exit code: always `0` unless the cvar database is broken.

## `diff` — compare two cfgs

```text
vanguard-config-builder diff A B
```

Parses two cfg files and reports cvar-level differences:

- `+ cvar = value` — present in B, absent in A
- `- cvar = value` — present in A, absent in B
- `~ cvar: old -> new` — value changed

Output is sorted by cvar name so two runs against the same inputs
produce identical output (good for `git diff`-style review).

### Example

```bash
$ vanguard-config-builder diff /etc/vanguard/server.cfg.before \
                               /etc/vanguard/server.cfg.after
# /etc/vanguard/server.cfg.before vs /etc/vanguard/server.cfg.after (3 change(s))
+ vanguard_dev = 1
- g_warmup = 30
~ sv_fps: 20 -> 40
```

Exit codes:
- `0` — comparison done (regardless of whether differences were found)
- `3` — file not found / unreadable

## Headless use cases

### Render config in Ansible

```yaml
- name: Render VanguardMod server.cfg from Cup profile
  ansible.builtin.command:
    cmd: >
      vanguard-config-builder generate
        --profile cup
        -o /etc/vanguard/server.cfg
        --strict
  notify: restart vanguardmod
  changed_when: true
```

The `--strict` default-on means a broken profile fails the playbook
loudly; the validator is your safety net before reload.

### Validate config in CI

A GitHub Actions step that fails the PR if the committed cfg has
problems:

```yaml
- name: Validate server.cfg
  run: |
    pip install vanguard-config-builder
    vanguard-config-builder validate ops/server.cfg --format json \
      | tee validation.json
    test "$(jq -r .valid validation.json)" = "true"
```

Exit `2` from `validate` propagates as a non-zero step result and
fails the workflow.

### Diff configs before deploy

Quick sanity check during a deploy script:

```bash
#!/usr/bin/env bash
set -euo pipefail

vanguard-config-builder diff /etc/vanguard/server.cfg \
                             /tmp/staged-server.cfg \
  | tee /tmp/cfg-diff.txt

read -rp "Apply changes? [y/N] " ans
[[ "$ans" == "y" ]] || exit 1

cp /tmp/staged-server.cfg /etc/vanguard/server.cfg
systemctl reload vanguardmod
```

### Render multiple profiles in one shot

```bash
for p in cup public scrim; do
  vanguard-config-builder generate \
    --profile "$p" \
    -o "/etc/vanguard/profiles/${p}.cfg"
done
```

## When to use which

| If you... | Use |
|---|---|
| Need a fresh `server.cfg` from a known profile | `generate` |
| Got an existing cfg from somewhere and want to sanity-check it | `validate` |
| Want to know what profiles are available | `list-profiles` |
| Want to know what cvars exist or filter by category | `list-cvars` |
| Want to review a config change before applying | `diff` |
| Want the GUI for interactive editing | run the binary with no subcommand |
